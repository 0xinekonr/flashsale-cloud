一、项目定位
────────────────────────────────────────────────────────────────────────────────
电商平台 + 高并发秒杀，以"生产级可落地"为标准，而非技术 demo。
核心业务场景：
  1. 普通下单：商品浏览 → 创建订单 → 锁库存 → 支付 → 完成
  2. 秒杀活动：10 万请求/分钟，库存 1000 件
     - 不超卖、不重复下单、可降级可限流、失败可补偿、最终一致
  3. 订单超时关闭：15 分钟未支付 → 自动取消 → 释放库存（延迟队列 + 定时扫描兜底）

后续可选扩展（不在主线路径）：
  - 优惠券/营销（coupon-service）
  - 通知服务（notify-service）
  - 报表服务（report-service）

  - coupon-service：优惠券发放/核销/规则引擎

四、事务策略（按场景选择，不一刀切）
────────────────────────────────────────────────────────────────────────────────
▸ ADR：为什么不在所有场景都用 Seata？
  Seata AT 模式通过全局锁实现隔离，在高并发写场景（秒杀）下全局锁竞争会成为瓶颈。
  因此针对不同并发量级选择不同策略：

┌──────────────────────────┬────────────────────────────────────────────────┐
│ 场景                     │ 方案                                          │
├──────────────────────────┼────────────────────────────────────────────────┤
│ 普通下单（中低并发）       │ 本地事务 + Outbox 模式                        │
│                          │ 订单写入 + outbox 消息在同一个本地事务中         │
│                          │ 异步轮询 outbox 表投递 MQ                       │
│                          │ 消费者端做幂等消费                              │
├──────────────────────────┼────────────────────────────────────────────────┤
│ 秒杀下单（高并发）        │ Redis 预扣 + MQ 异步 + 最终一致                │
│                          │ 不走 Seata，不走分布式事务                      │
│                          │ 库存在 Redis 原子扣减，订单异步创建              │
│                          │ 超时取消 → MQ 通知库存回滚                      │
├──────────────────────────┼────────────────────────────────────────────────┤
│ 支付回调                  │ 幂等回调 + 本地事务                            │
│                          │ 验签 → 幂等检查 → 更新支付状态 → 发 MQ          │
│                          │ 不需要分布式事务                                │
├──────────────────────────┼────────────────────────────────────────────────┤
│ 补偿/对账                 │ 定时扫描 + 自动修复                            │
│                          │ 兜底机制，处理所有边界 case                     │
└──────────────────────────┴────────────────────────────────────────────────┘

================================================================================
M2：一致性基础设施（Outbox + 幂等框架）
================================================================================

  ▸ 这是整个项目的"骨架级"基础设施，必须在秒杀完善之前就位。

  M2.1 common-mq 模块
    - 新建 libs/common-mq 模块
    - OutboxMessage 实体：id, aggregateType, aggregateId, eventType,
      payload(JSON), status(PENDING/SENT/FAILED), createTime, sentTime
    - OutboxPublisher：在业务事务中写入 outbox 表（同一本地事务）
    - OutboxPoller：@Scheduled 定时扫描 PENDING 消息，投递到 RabbitMQ，
      投递成功后标记 SENT
    - IdempotentConsumer：消费者装饰器，基于 messageId 查 idempotent_record 表
      去重，消费成功后写入记录

  M2.2 common-redis 模块
    - 新建 libs/common-redis 模块
    - IdempotentKeyInterceptor：基于 HTTP 请求的幂等拦截器
      （读取 X-Idempotent-Key header，Redis SETNX 实现，TTL 可配）
    - DistributedLock：基于 Redisson 的分布式锁工具

  M2.3 order-service 接入 Outbox
    - 创建订单时：本地事务中同时写 orders 表 + outbox 表
    - outbox 消息类型：ORDER_CREATED（通知 inventory lockStock）
    - outbox 消息类型：ORDER_CANCELLED（通知 inventory unlockStock）
    - 移除 createOrder 中的同步 Feign 调用 lockStock
      （改为：写 outbox → 异步投递 → inventory 消费 → 回调确认）

  M2.4 inventory-service 接入幂等消费
    - lockStock 消费者使用 IdempotentConsumer 包装
    - 确保同一 orderId 的 lockStock 请求只执行一次

  验收：
    ✓ 创建订单后，outbox 表有 PENDING 记录
    ✓ 3 秒内 outbox 消息被投递到 MQ（状态变为 SENT）
    ✓ 杀掉 RabbitMQ → 创建订单仍成功 → 重启 RabbitMQ → 消息最终投递成功
    ✓ 同一消息重复投递，inventory 只执行一次 lockStock

M4：支付完善 + 对账补偿
================================================================================

  M4.1 支付服务生产化
    - 模拟支付签名验证（MD5/HMAC 签名 + 密钥，模拟支付宝异步通知格式）
    - 支付回调幂等：基于 transactionId 唯一索引 + 状态检查
    - 支付超时处理：支付创建后如果订单已取消，拒绝支付
    - 退款流程（订单取消时如果已支付，自动发起退款）

  M4.2 对账补偿任务
    - order-service 定时扫描：
      每 5 分钟扫描 status=NEW 且 createTime > 20 分钟的订单
      （延迟队列的兜底，防止 MQ 故障导致超时取消丢失）
    - inventory-service 对账：
      每 30 分钟比对 lockedStock 与实际 NEW 状态订单数
      不一致时报警 + 自动修复

  M4.3 超时/重试/降级策略文档化
    - Feign 超时配置：connectTimeout=2s, readTimeout=5s
    - 重试策略：只有 GET 请求重试，POST 不重试（防止重复写）
    - Sentinel 降级：秒杀服务 → 静态排队页；库存服务 → 返回锁定失败
    - 配置写入 Nacos，可动态调整

  验收：
    ✓ 伪造签名的支付回调被拒绝（返回 403）
    ✓ 同一支付回调重复发送 3 次，订单状态只变更 1 次
    ✓ 手动杀掉 RabbitMQ → 创建订单 → 15 分钟后定时扫描兜底取消
    ✓ 对账任务检测到不一致时输出告警日志

八、安全基线（贯穿所有阶段）
────────────────────────────────────────────────────────────────────────────────

  ✓ SQL 注入：MyBatis Plus 参数化查询，不拼接 SQL
